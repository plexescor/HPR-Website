UI Hacking Guide

Hack the HPR User Interface

HPR doesn't just let you swap colors. By loading Slint layouts dynamically at runtime, you can completely restructure the UI, build custom panels, and tweak animations without compiling a single line of C++.

Get Started → Folder Layout The UI Contract
Architecture

How the Theme System Works

Normally, HPR compiles its user interface directly into the executable for absolute minimum memory usage. However, by enabling Interpreter Mode, HPR bypasses the compiled-in interface. Instead, the C++ backend spins up Slint's runtime compiler, parses layout files from your config directory, and mounts them live.

💡 The Performance Trade-Off

Dynamic compilation has a memory cost. When running in Interpreter Mode, HPR's private RAM usage climbs from its ultra-light compiled footprint of **~12MB** up to **~110MB (a 20-50% increase)**. This is due to Slint's AST parsing tables, bytecode interpreter, and decoded preview image buffers sitting in memory. It's totally fine for daily use, but it's good to keep in mind!

⚙️

1. Turn on Interpreter Mode

Open HPR's global configuration file and enable interpreter loading:

config.csv 
use-interpreter,true
📂

2. Locate your themes folder

HPR scans a dedicated themes subdirectory inside your config root:

Paths 
Windows: %APPDATA%\HPR\HPR_Config\themes\
Linux:   ~/.config/HPR/themes/
Walkthrough

Creating Your First Theme

Let's build a custom theme called neon-cyber from scratch. Follow these steps:

Step 1: Set up the folder

Go to your HPR themes/ directory and create a new folder named exactly after your theme: neon-cyber.

Step 2: Declare Metadata

Inside neon-cyber/, create a file called metadata.csv. This file registers your theme name and matches it with the HPR engine version. Key-value pairs are separated by a comma:

themes/neon-cyber/metadata.csv 
# Comments start with #
name,Neon Cyber
version,0.9.2

💡 Version Checks: If your theme's version doesn't match the current HPR binary version, the engine will still load it, but it will render a warning badge in the Themes tab alerting you that it might be out of date.

Step 3: Write the Layout

Create app-window.slint inside your theme folder. This file represents your entire user interface layout. Instead of writing it from blank paper, always copy HPR's reference files first so you have the correct structure:

HPR writes clean default layout files to ui-REFERENCEONLY/ in your application directory on every launch. Copy app-window.slint and its dependencies (like card.slint, palette.slint, etc.) directly into your neon-cyber/ directory. Now you can edit styles, modify layouts, or re-arrange buttons!

Step 4: Bundle Previews (Optional)

HPR shows scrollable screenshots of your design inside the theme chooser tab before users click Apply. You can place up to 9 screenshots named 1.png, 2.png, ..., 9.png in your theme folder.

⚠️ The Chain Rule: HPR scans preview files sequentially. If you skip a number, HPR stops loading pictures immediately. For example, if your folder has 1.png, 2.png, and 4.png, HPR will only display 1 and 2 because it hit a wall at 3.png.

Step 5: Load it!

Launch HPR, head to the Themes section, click RESCAN to let the engine pick up your folder, select your theme in the dropdown, and hit APPLY. The interface will instantly reload with your modifications!

Critical Rules

The C++ / Slint Contract

HPR's compiled C++ engine communicates with the dynamic Slint UI by mapping specific properties and callbacks by name. If you delete or rename these, the backend will fail to bind them and your interface will break.

MainWindow Component The main root component in your Slint file must be exported and named exactly MainWindow. This is what the C++ engine searches for to instantiate the window.
active-view Property Tracks which sidebar section is currently open (e.g. "HOME", "THEMES"). When the user navigates away from the themes tab, C++ reads this property and unloads large theme screenshot assets from RAM automatically to keep memory clean.
useThemesEnabled_S / themesAvailable_S Booleans pushed by the engine to tell the UI whether the theme dropdown and settings toggle should be active.
refreshThemes() Callback Must be bound to your "Rescan" button. Tells the C++ backend to scan your themes directory and refresh the dropdown options.
themeSelected(string) Callback Must be triggered when a theme is selected in the list/dropdown. Tells the C++ engine to reload the preview image list in the background for that theme.
themeApply(string) Callback Must be triggered when the user clicks "Apply". Instructs the backend to store the new theme choice in config.csv and hot-reload the UI engine path.
💡 Tip: Restore Default UI

If a theme breaks or throws binding warnings, select the **default** theme option in the UI. HPR will reset the interpreter path and reload the safe reference UI files from ui/app-window.slint.
